# 数据处理问题（训练前阻断清单）

## 结论

当前数据处理链路可以重跑并复现现有产物，但在“用于模型训练且不能出现任何错误”的标准下，**暂不满足直接开训条件（BLOCK）**。

本文件仅记录已确认问题与风险，不包含代码修改。

---

## 1) 阻断问题（Blocking）

### 1.1 同日 Baidu 原始列仍在特征数据中，存在误用即泄漏风险

- 现象：`park_featured_data.csv` 中同时存在
  - `baidu_PC+移动指数`（同日原始值）
  - `baidu_PC+移动指数_lag1d`（滞后1日）
- 风险：若训练脚本按“全数值列自动入模”，会把同日原始 Baidu 一起喂入模型，违反“同日不可见”的泄漏约束。
- 影响：训练指标可能虚高，线上泛化失真。

### 1.2 特征窗口语义与真实采样节奏不一致

- 现象：`build_features.py` 注释将 `96` 行解释为“约1天（15分钟频率）”。
- 实测：当前 park 数据为**每日固定 48 行**（09:00–20:45），跨日间隔为 12h15m，不是 24h 连续分钟序列。
- 风险：
  - `ma_96 / vol_96 / slope_96` 等并非“1天窗口”语义；
  - `baidu_ma_96 / baidu_ma_288` 也会被误解为错误时间跨度。
- 影响：特征解释与研究结论可能错位，影响实验可解释性与决策可靠性。

### 1.3 当前校验以抽样为主，缺少训练输入层面的硬门禁

- 现象：`merge_daily_data.py` 与 `validate_daily_pipeline.py` 都包含随机抽样校验（固定 seed），能发现部分错误，但非训练级“强约束”。
- 缺口：
  - 没有“禁止列/允许列”训练门禁；
  - 没有把“时点可得性”规则落为可执行阻断条件。
- 风险：即使数据文件本身一致，仍可能在训练入口误用未来信息或不该可见列。

## 2) 已通过的检查（事实）

以下检查已完成且结果正常：

- `validate_daily_pipeline.py` 可运行通过；
- `park_aligned_data.csv` 与 `merge_daily_data.py` 重建结果全量一致；
- `park_featured_data.csv` 与 `build_features.py` 重建结果在数值容差下全量一致；
- 日期连续：`2022-11-18` 到 `2025-12-31` 无缺失天；
- 每天行数恒为 48；
- 日级源（weather/holiday/baidu）在该日期范围内均为唯一日期、无缺失覆盖；
- `lag1d` 严格满足“首日 NaN，后续等于前一日 Baidu 值”。

这些结果说明“当前产物与当前代码一致”，但不等于“训练无泄漏风险”。

## 3) 非阻断但应修复项（Non-blocking）

- 静态类型告警存在（`build_features.py` 与 `merge_daily_data.py` 有 Pyright 报告）。
- 这类问题当前更偏类型推断/签名层，不是已证实的数据错误来源；但建议在训练前清理，以降低后续维护与重构风险。

---

## 4) 训练放行前最小条件（Go-live Minimum）

在满足以下条件前，不建议开始正式训练：

1. 明确预测时点与可用信息边界（哪些变量在该时点“真实可知”）。
2. 在训练入口强制特征白名单，显式排除同日 Baidu 原始列及其他不满足时点可得性的列。
3. 将窗口语义与真实采样节奏对齐（按 48 行/天解释或重定义），避免“名义1天/实际2天”错配。
4. 将上述规则落成自动阻断检查（不满足即 fail），而不是仅靠抽样验证。
